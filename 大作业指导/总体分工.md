这个作业本质上是一个“开源项目漏洞挖掘 + 软件度量”期末大作业。

你们要从文档附录给出的 20 个真实项目里选至少 3 个项目，对每个项目都做：

1. 动态测试：比如 AFL、libFuzzer、honggfuzz 等模糊测试。
2. 静态分析：比如 Clang Static Analyzer、Cppcheck 等。
3. 记录过程和结果：环境、工具、driver/fuzzer、运行截图、覆盖率趋势、静态分析截图。
4. 如果发现 bug：说明位置、类型、原因、危害，最好能复现并提交 issue。
5. 可选加分：设计 Agent 工具流，比如让 Agent 帮忙生成测试用例、分析 crash、辅助 PoC 复现等。

最后交两样东西：

- 10 分钟口头报告 PPT，在第 14-16 周报告。
- 不超过 10 页的实验报告，第 16 周提交。

报告里必须有每个项目的：

- 手写 driver/fuzzer 截图；
- 跑 12 小时 fuzzing 的截图或覆盖趋势图；
- 静态分析工具运行截图。

**4 人分工建议**

我建议按“3 个项目负责人 + 1 个统筹/验证负责人”来分，而不是按“一个人只做 PPT、一个人只写报告”。因为评分重点在真实漏洞、测试工具有效性、工作量和报告质量，所以每个人都要有技术产出。

**成员 A：项目 1 负责人**

负责一个相对容易上手的库，比如 `cJSON`、`zlib`、`libpng`、`libexpat` 这类输入格式明确的项目。

主要工作：

- 搭建编译环境；
- 写 libFuzzer 或 AFL driver；
- 准备初始语料库；
- 跑至少 12 小时 fuzzing；
- 记录 crash、覆盖率、截图；
- 做该项目的静态分析。

**成员 B：项目 2 负责人**

负责第二个项目，最好和 A 类型不同，比如 `libpcap`、`libsndfile`、`libxlsxwriter`、`libplist`。

主要工作同上，但重点可以放在不同输入格式上，比如 pcap、音频文件、xlsx、plist 等。

**成员 C：项目 3 负责人**

负责第三个项目，可以选稍微复杂一点、体现工作量的项目，比如 `curl`、`libvpx`、`libgd`、`libass`。

主要工作：

- 找到适合 fuzz 的入口；
- 从项目自带测试、examples、CLI 工具中改 driver；
- 跑 fuzzing；
- 配合 ASAN/UBSAN 定位问题；
- 整理该项目的漏洞或“未发现漏洞原因”。

**成员 D：统筹、复现、Agent 与报告负责人**

这个人不是只写文档，而是做横向支撑：

- 统一 fuzzing 环境，比如 Docker、Ubuntu 虚拟机、AFL++、clang、sanitizer 版本；
- 统一截图规范和文件夹结构；
- 帮 A/B/C 复现 crash，确认是不是项目本身 bug；
- 负责静态分析结果去重和误报判断；
- 设计 Agent 加分部分，例如：
  - 让 Agent 阅读 API/example 生成 fuzzer 初稿；
  - 让 Agent 分析 crash stack trace；
  - 让 Agent 根据静态分析告警生成测试输入；
  - 让 Agent 辅助写 PoC 和 issue 描述；
- 汇总实验报告和 PPT。

**推荐执行节奏**

第 1 阶段：选题和环境，1-2 天  
先选 3 个项目，不要都选巨型项目。建议组合是：

- 稳妥型：`cJSON` / `zlib` / `libpng`
- 中等型：`libpcap` / `libexpat` / `libsndfile`
- 挑战型：`curl` / `libvpx` / `libgd`

第 2 阶段：每个项目先跑通最小 fuzzer，2-4 天  
目标不是一开始就发现漏洞，而是先做到“能编译、能喂输入、能跑起来、能出覆盖率”。

第 3 阶段：正式跑 12 小时，3 个项目并行  
每个项目都保留：

- 命令行截图；
- AFL/libFuzzer 运行界面截图；
- 覆盖率趋势图或 `plot_data`；
- crash/hangs 目录；
- 静态分析输出截图。

第 4 阶段：漏洞确认和报告，2-3 天  
有 crash 就用 ASAN/gdb 复现，确认堆栈、输入、触发路径。没有 crash 也要解释：测试入口覆盖了哪些 API、为什么可能没发现、后续如何改进。

**比较合理的最终交付结构**

实验报告可以按这个结构写：

1. 任务概述和小组分工
2. 测试环境与工具
3. 项目一：对象、driver、动态测试、静态分析、结果
4. 项目二：同上
5. 项目三：同上
6. Crash/漏洞分析或未发现原因
7. Agent 工具流设计
8. 总结与改进

PPT 则控制在 8-10 页左右：

- 选题和方法；
- 三个项目各 1-2 页；
- 发现的 bug 或典型 crash；
- Agent 加分设计；
- 总结。

一句话概括：这不是普通读论文作业，而是要真的把开源项目跑起来、写 fuzzer、跑静态分析、尽量挖 bug，并把全过程证据整理成报告。4 个人做的话，最稳的是 3 人各扛一个项目，1 人做环境、复现、Agent 和最终集成。
